# Rust Test Support File Pattern

Keep test-only code out of production source files while maintaining cohesion and privacy.

## Problem

Production code sometimes needs to call test-specific logic (e.g., mock handlers, test fixtures). Inline `#[cfg(test)]` blocks pollute production files with test code.

## Solution

Use `#[path]` directive to conditionally include test support from a separate file:

```
src/module/
├── foo.rs                 # Production code (clean)
├── foo_test_support.rs    # Test helpers (separate file)
├── tests.rs               # Unit tests
└── mod.rs
```

### Production file (foo.rs)

```rust
// Only 4 lines of test boilerplate in production file
#[cfg(test)]
#[path = "foo_test_support.rs"]
pub(crate) mod test_support;

fn production_function() {
    // ...
    
    #[cfg(test)]
    if needs_test_handling {
        return test_support::test_handler();
    }
    
    // ... production logic
}
```

### Test support file (foo_test_support.rs)

```rust
//! Test support for foo module.
//!
//! This file is only compiled during tests via #[path] include.

use std::collections::HashSet;

/// Test-only helper function.
/// 
/// Use pub(crate) so sibling test modules (tests.rs) can access.
/// Use pub(super) if only the parent module needs access.
pub(crate) fn test_handler() -> HashSet<String> {
    // Test implementation
    HashSet::new()
}

pub(crate) fn parse_test_data(input: &str) -> ParsedData {
    // Test parsing logic
}
```

### Unit tests (tests.rs)

```rust
use super::foo::test_support::{test_handler, parse_test_data};

#[test]
fn test_handler_returns_empty_set() {
    let result = test_handler();
    assert!(result.is_empty());
}
```

## Visibility Guide

| Visibility | Use When |
|------------|----------|
| `pub(super)` | Only parent module (foo.rs) needs access |
| `pub(crate)` | Sibling modules (tests.rs) also need access |

## Benefits

| Aspect | Outcome |
|--------|---------|
| Production file | Clean—only 4 lines of `#[cfg(test)]` boilerplate |
| Separation | Test code in dedicated `*_test_support.rs` file |
| Privacy | `pub(crate)` keeps functions crate-private (test builds only) |
| Cohesion | Support file sits adjacent to production file |
| Discoverability | Naming convention makes relationship clear |

## When to Use

- Production code needs conditional test behavior
- Test helpers are substantial (>20 lines)
- You want to minimize context window load when reading production code

## When NOT to Use

- Simple `#[cfg(test)]` constants or small helpers (<10 lines)
- Test code that doesn't need to be called from production code
- Use regular `foo.test.rs` pattern for pure unit tests instead

## Real-World Example: Commutative Merge

The `src/orchestration/aggregate/` module uses this pattern for test state diffing:

**Problem:** `diff_state_fields()` in merge.rs needs to handle `test.StatefulState` type during tests using JSON-like parsing. This required ~100 lines of test-only code.

**Before:** merge.rs contained inline `#[cfg(test)]` functions and a test module—readers had to mentally skip test code when understanding the production logic.

**After:**
```
src/orchestration/aggregate/
├── merge.rs                 # 205 lines - production only
├── merge_test_support.rs    # 55 lines - test helpers
├── tests.rs                 # Unit tests import from test_support
└── mod.rs
```

**merge.rs** (clean production file):
```rust
#[cfg(test)]
#[path = "merge_test_support.rs"]
pub(crate) mod test_support;

pub(crate) fn diff_state_fields(before: &Any, after: &Any) -> HashSet<String> {
    // ...
    
    #[cfg(test)]
    if before.type_url == "test.StatefulState" {
        return test_support::diff_test_state_fields(&before.value, &after.value);
    }
    
    // ... production logic (proto_reflect, byte fallback)
}
```

**merge_test_support.rs** (test helpers):
```rust
//! Test support for merge module.
//! Only compiled during tests via #[path] include in merge.rs.

pub(crate) fn parse_test_state_fields(s: &str) -> HashMap<String, String> { ... }
pub(crate) fn diff_test_state_fields(before: &[u8], after: &[u8]) -> HashSet<String> { ... }
```

**tests.rs** (unit tests):
```rust
use super::merge::test_support::{diff_test_state_fields, parse_test_state_fields};

#[test]
fn test_diff_test_state_fields_single_change() {
    let before = r#"{"field_a":100,"field_b":200}"#.as_bytes();
    let after = r#"{"field_a":100,"field_b":300}"#.as_bytes();
    let changed = diff_test_state_fields(before, after);
    assert!(changed.contains("field_b"));
}
```

**Result:** Production file reduced from ~300 lines to ~205 lines. Test code cleanly separated but still co-located for discoverability.
